# CLAUDE.md — 本仓库不可协商的工作铁律

> 本文件每个会话自动加载。以下铁律的优先级 **高于** 我上下文里的任何惯性记忆、
> 经验直觉或"我觉得应该是这样"。冲突时，一律以铁律为准。

## 铁律 1 —— Skill / ROUTINE 优先于惯性

- 每次开工、以及任何一次要动手采集/编辑报告/改站点之前，**先读 `ROUTINE.md` 全文**，
  按当前版本规则执行。不得凭记忆里的旧规则行事。
- 若存在与当前任务相关的 Skill，**必须先调用该 Skill**，再动手。Skill/routine 的
  指令优先级高于我脑内的惯性做法。
- "我记得规则大概是……"不构成跳过阅读的理由。规则以仓库文件的**当前版本**为唯一事实源。

## 铁律 2 —— 出错必先读日志原文，严禁凭空推测

- 任何执行错误（CI 失败、部署失败、脚本报错、命令非零退出），**第一步永远是读取该错误
  对应的 log 原文**（如 `get_job_logs` 取真实 stderr / 步骤日志），定位 `##[error]` 原句。
- 在读到真实报错原文之前，**禁止**给出"大概是缓存 / 大概是网络 / 大概是后端抽风"之类的
  经验性推断，也禁止据此采取补救动作。
- 找到根因后，处理步骤必须**严谨可复现**：说明是哪一行错误、根因是什么、为什么这个修法能解决。
- **不甩锅**：即便根因是外部服务（如 Pages 后端间歇失败），也必须给出确定性的应对步骤
  （见下方部署铁律），不得以"外部抽风"作为不作为的借口。

## 铁律 3 —— GitHub Pages 部署（血泪教训）

本仓库用 `.github/workflows/deploy.yml` 单-job 部署（upload-pages-artifact + deploy-pages 同 job）。

- **失败后严禁用 `rerun_failed_jobs` 重跑该 job**：单-job 结构下重跑会再上传一份同名
  `github-pages` artifact，叠加成 2 份 → `deploy-pages` 直接报
  `Multiple artifacts named "github-pages"` 秒失败。这是**自制的第二种失败**。
- 后端出现 `Deployment failed, try again later`（间歇性、约 50%）时，正确应对是
  **触发一个全新 run**（`workflow_dispatch` on `main`），而不是重跑旧 run——
  新 run 有干净的 artifact 空间，且能碰上后端好窗口。
- 每次判定"上线成功"前，必须确认对应 workflow run 的 `conclusion == success`，
  并提醒用户**强制刷新**（浏览器/CDN 缓存会让首页看起来没更新）。

## 铁律 4 —— 报告发布前做跨区一致性自检（ROUTINE v5）

- 每日报告**定稿、推送前**，必须**通读全页**，逐条核对精选↔扩展↔噪音三区之间
  有无**同源矛盾**（同一底层事实以相反价值判断出现在两个区）。
- 同一事实只能落一个区；夸大说法若是对某精选条目的曲解，**并入该精选卡「坑与验证」**，
  不得另立噪音条。发现矛盾先合并/删除，确认无误后在页脚写明"已做跨区一致性自检"再发布。
- 依据：LLM 分段填格子、不会自发全页审校——把此检查显式编码为**强制收尾步骤**，不靠自觉。
  细则见 `ROUTINE.md` §5（此规则自 v5 引入；ROUTINE **当前版本 v7**——v6 为页面设计改版（br- 设计系统、
  全站去 emoji、`docs/style.css`、锚点 `#topic-<id>`、旧文件备份 `legacy-v1/`）；v7 新增发布门禁与部署分工，见铁律 5）。

## 铁律 5 —— 发布门禁与部署分工（2026-07-13 用户授权）

- **推送前必须 `python3 scripts/validate.py` 通过（退出码 0）**：结构 / 锚点 / 去 emoji / 精选卡模板
  四字段与字数下限 / `links[]` 原始信源非空。失败 = 禁止发布，先修再推。
- **内容型提交**（`docs/reports/*.html`、`docs/data/topics.json`）：校验通过后**直接 push `main`**（Pages 自动部署），
  这是用户明确授权的常态路径——**不得**再以"分支规则"为由把每日例行内容滞留在分支（2026-07-13 教训：
  分支规则被误泛化，导致当日内容未上线；7/7-7/12 的每日例行一直是直推 main）。
- **工程 / 规则改动**（index.html、style.css、scripts/、ROUTINE.md、CLAUDE.md、workflow）：**走 PR** 由用户审阅。
- 推 main 后必须按铁律 3 确认 deploy run `conclusion == success`，并提醒用户强刷缓存。
