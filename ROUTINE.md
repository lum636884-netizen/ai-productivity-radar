# ROUTINE — AI 生产力雷达 信息收集与交叉验证规则

**当前版本：v5**（2026-07-09 生效）

> 术语约定：本系统定位为面向个人生产力的 **AI 能力信息收集系统**。所有页面、文档、提交信息使用"信息/资讯"类表述。

## 0. 使命

面向"一人团队"（印尼公共充电场站业务：加盟商招商 + Marketing）的每日 AI 生产力信息。
每条信息必须回答：这对我的工作流意味着什么、要不要动手、怎么动手。
严格过滤噪音，宁缺毋滥。

## 1. 信源分级（决定是否纳入）

优先级：**T2 > T1 > T3**（可实践落地优先于反应速度）

- **T2 可复现实践**：附完整代码/配置/操作步骤的帖子、技术博客、GitHub 仓库。可信度看"能否照着做出来"。
- **T1 官方一手**：官方 changelog、官方文档更新、GitHub release notes、官方工程博客。本身即事实源，无需交叉验证。
- **T3 观点与传闻**：KOL 推文、YouTube 解读、无实操细节的评价类内容。**默认过滤**。仅当多个独立 T3 信源指向同一事件、且能回溯至 T1/T2 时才纳入。

**热度只用于同级内容排序，不用于决定纳入。**

## 2. 纳入白名单（命中其一才进报告）

1. **能力变化**：新模型、新功能、参数与行为变更（影响现有工作流的）
2. **一人团队可部署实操**：营销自动化、内容生产、招商线索处理、行研工作流、编程 agent 协作模式
3. **可靠性工程**：降低幻觉、长程任务稳定性、交付物验证方法、评估手段
4. **踩坑警报**：社区集中报告的 bug、性能退化、误导性宣传的辟谣

**排除项**（即使高热度也不进）：AI 行业投资/融资、股价变动、AI 机器人硬件、
与一人团队生产力无关的纯学术进展。

**加分项（不占精选名额，放扩展区）**：与充电场站/东南亚市场直接相关的 AI 营销、招商应用案例。

## 3. 置信标签

- ✅ **已证实**：官方来源直接确认
- 🔶 **可复现**：有完整步骤/代码，未经官方确认
- ⚠️ **待观察**：多信源提及但缺一手证据 → 明确标注"先别投时间"

## 4. 卡片结构（每条精选信息）

> 是什么 → 对我的场景值多少（明确到招商/Marketing 一人团队）→ 怎么部署（步骤或指向 T2 源）→ 坑与验证方法 → 行动建议（今天就试 / 观望一周 / 忽略）

## 5. 数量与版面

- 精选卡片：**单次运行 3-5 条**，宁缺毋滥。无达标信息时明确写"今日无值得行动的更新"。
- 扩展区：其他有价值但未入精选的条目，简要罗列，与精选区明确区隔。
- 今日噪音：1-2 条被过滤的高热度夸张内容 + 过滤理由。

**跨区去重铁律（v5 新增，血泪教训）：**
- **同一底层事实只能落一个区**（精选/扩展/噪音三选一）。禁止把同一条信息以相反的价值判断同时放进两个区。
- 若某个夸大说法是对某精选条目的**曲解/残缺版**，把它**并入该精选卡的「坑与验证」**作为辟谣，**不得另立噪音条**——噪音区只收与当期精选/扩展**无实质重叠**的独立条目。
- 若两条同主题但**价值判断一致**（如"用官方审核目录"与"别碰无审核海量目录"、"成本偏高"与"服务抖动"皆为审慎方向），可分置不同区，但须确认方向一致、非矛盾。

**发布前一致性自检（v5 新增，强制收尾步骤）：**
- 定稿、推送前**通读全页**，逐条核对精选↔扩展↔噪音之间是否存在**同源矛盾**（同一事实、相反价值）。
- 发现矛盾→按上条合并/删除处理；确认无误后，在报告页脚写明"**已做跨区一致性自检：精选/扩展/噪音无同源矛盾条目**"再发布。
- 依据：LLM 分段填格子、不会自发做全页全局审校，故必须把此检查显式编码为强制步骤，不靠自觉（同"时区前置计算""部署铁律"一套打法）。

**同日多次运行（追加制，每天一个入口）：**
- 同一日期若执行多次，第 N 次运行**不覆盖、不另建文件**，将新内容**追加写入当日已有 HTML 文件**。
- 追加位置：新精选卡片追加至精选区末尾，新扩展条目追加至扩展区末尾，新噪音条目追加至噪音区末尾。每次追加前插入时间戳分隔线（如 `── 第 2 次采集 · HH:MM（UTC+8）──`），区分批次。
- 页面顶部统计行同步更新（如"精选 5 条 · 本日已采集 2 次"）。
- **数量规则按单次运行计**：3-5 条是单次运行的质量锚，不是全天上限。
- 每次追加运行独立执行完整的信源分级、白名单筛选与交叉验证，且不得与当日已推内容重复（同事件有实质新进展除外，标注"更新"）。

## 6. 链接保留标准（每条 3-5 个）

优先级：
1. 可复现步骤最完整的实操源
2. 官方一手源
3. 含质疑与验证过程的高质量讨论串

不保留：营销号转述、无增量的二手总结。

## 7. 推送与时区

- **时区铁律：所有日期判断（报告文件名、"当日/昨日"界定、周日复盘触发）一律按 UTC+8 换算后确定，禁止直接使用系统 UTC 日期。**（v2/v3 时期两度因此产生文件归属错误，已归并修复）
- **前置计算（每次运行第一步，先算再写）：** 取当前实际时间戳（系统时钟 / workflow 触发时间的 UTC）→ `+8h` 得 UTC+8 → 得出当日日期 `YYYY-MM-DD` 与星期。**先在对话里明确写出"本次归属日期＝UTC+8 X 月 X 日（周几）"，再动手写任何文件。** 禁止沿用上一次会话/上一批的日期惯性，禁止凭印象判断"今天是几号"。跨零点、跨批次尤其复查——UTC 02:34 = UTC+8 10:34 属次日。
- **路径铁律：** 任何指向报告的 URL/链接，一律为 `.../ai-productivity-radar/reports/YYYY-MM-DD.html`（含 `reports/` 段）。对外发路径**从 `topics.json` 的 `days[].url` 复制**，或按本规则拼接，**禁止凭记忆手写省略 `reports/`**。发送前对照 `docs/reports/` 实际文件名核对一次。
- 每日 UTC+8 早 8:00 前生成完毕（当日追加运行不受此时限）
- 中文撰写，保留原文链接与关键英文术语
- HTML 发布至 GitHub Pages，**每天仅一个文件**：`docs/reports/YYYY-MM-DD.html`（UTC+8 日期）。同日追加运行更新同一文件，不产生后缀文件。
- 报告页面使用共享样式表 `docs/assets/style.css`（外链，不内嵌 CSS）；每张精选卡片带锚点 id（供首页与数据层深链）。
- docs/index.html 为数据驱动页面（见第 11 条），**不手工编辑**。
- 在对话中发送当日 HTML 文件
- 页脚标注当期 ROUTINE 版本号

## 8. 打磨机制

1. **随手反馈**：对话中任何反馈即触发规则修订并升版本号
2. **漏报归因**：用户提供漏报案例 → 逆向分析（信源未覆盖 / 规则误杀）→ 修规则
3. **每周复盘**：每周日（UTC+8）报告末尾附本周"行动建议"命中率回顾
4. **变更记录**：所有规则改动记录于下方 Changelog，可审计、可回滚

## 9. 已推内容时效规则（滚动窗口）

**目标**：避免同一条信息反复占用精选名额，同时不漏掉仍在行动窗口内的重要更新。

**默认窗口：7 天。** 一条信息进入精选后，7 个自然日内不再重复精选，除非触发下列任一**再入选条件**：

1. **反转或重大修正**：新的 T1/T2 源与原精选结论直接矛盾（如官方撤回、已修复的 bug、定价调整）
2. **Deadline 临近**：原精选中有明确截止日期，且距截止 ≤ 3 天时，可重新入选并标注剩余天数
3. **实质性新进展**：有显著新 T1/T2 证据补充原有信息（如"观望一周"后有了明确结论）

**扩展区不受此限制**：已推内容的后续小更新可在扩展区用一行注明"后续更新"，不占精选名额。

**执行方式**：每次运行前，读取 `docs/data/topics.json` 中近 7 天条目，标注窗口内的话题，再对比本次候选内容。

## 10. 信息采集深度规范

**目标**：消除不同模型/不同运行之间搜索深度不一致导致的漏报。

**社区来源定义（含社交媒体）**：Reddit（r/ClaudeAI 等）、Hacker News、X/Twitter（官方账号 + 工程师）、LinkedIn 行业讨论、Discord 官方频道、Substack/技术博客。

**最低要求（每次运行，指发起搜索的次数，非结果条目数）**：

- 独立搜索请求（WebSearch + WebFetch 合计）**不少于 6 次**，覆盖以下全部类型：

| 类型 | 次数 |
|------|------|
| 官方 changelog / release notes（优先 WebFetch 官方源，勿只靠搜索摘要） | 1-2 |
| 能力变更 + 新模型动态 | 1-2 |
| 社区实践 + 踩坑（含社交媒体） | 1-2 |
| 弃用 / 截止日期通知 | 1 |
| 社交媒体热议交叉验证 | 1 |
| 场景加分项（充电场站 / 东南亚 / 招商营销） | 1（可选） |

**追挖规则**：若 T3 来源提及某事件且命中纳入白名单，必须追加至少 1 次搜索寻找 T1/T2 确认；找不到则降级至扩展区并标 ⚠️ 待观察。

**不追挖的情形**：官方 T1 直接证实的条目、纯噪音条目（直接过滤）。

## 11. 结构化数据层（topics.json）

**单一数据源**：`docs/data/topics.json`。首页的全部动态内容（30 天精选、热词、分类筛选、搜索、话题列表、每日报告列表）均由它渲染，**首页 HTML 本身不手工维护内容**。

**每次运行的数据义务**（与写 HTML 报告同步完成）：
1. 每条精选/扩展话题追加一条记录：`{id, date, batch, tier(featured/extended), cat(主分类), conf(置信), act(行动建议), deadline, title, brief, tags[], url(报告锚点), links[]}`
2. **标签在生成时打好**，不事后由 AI 重新推断（避免漂移）
3. 重算 `picks`（30 天精选）与 `hot`（热词），更新 `days`（每日报告列表）与 `updated` 时间戳
4. picks/hot 每次基于全量数据幂等重算，不依赖上次中间状态

**主分类（7 类，稳定集，不轻易增改）**：

| key | 名称 |
|-----|------|
| model | 🧠 模型能力 |
| tools | 🛠 工具与工作流 |
| marketing | 🎯 营销与获客 |
| reliability | 🛡 可靠性工程 |
| cost | 💰 成本与计费 |
| pitfall | ⚠️ 踩坑与截止 |
| platform | 🔐 平台与合规 |

一条话题一个主分类 + 若干自由 tags（tags 供搜索与热词关联）。

## 12. 首页信息架构

**① 30 天精选（置顶）**：从 topics.json 取最近 30 天全部话题，按精选标准（白名单命中度 + 置信度 + 行动价值 + 是否仍在行动窗口）提炼 **5-10 条**，可合并同主题话题为一张卡；对入选条目补全信息逻辑、优化表达，必要时按第 10 条规则联网补搜。视觉上与每日列表区分（独立配色置顶区块）。30 天滚动，过期话题自动滑出。

**② 分类筛选 + 搜索**：7 个分类 chip + 全文搜索框（标题/摘要/标签），纯客户端执行。

**③ 热词（与分类正交，时效性）**：
- **入选**：30 天内被 ≥2 条话题引用的关键词，或单条高价值截止/破坏性事件
- **展示**：独立区域，带引用次数与剩余天数，可点击筛选
- **移除**：连续 **14 天**无新话题引用 → 自动降温移除；关联截止日过去 → 移除（前端按 expire 字段自动隐藏）；被后续 T1/T2 证伪或官方修复 → 数据更新时移除
