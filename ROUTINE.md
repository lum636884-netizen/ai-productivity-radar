# ROUTINE — AI 生产力雷达 交叉验证规则

**当前版本：v2**（2026-07-05 生效）

## 0. 使命

面向"一人团队"（印尼公共充电场站业务：加盟商招商 + Marketing）的每日 AI 能力情报。
每条信息必须回答：这对我的工作流意味着什么、要不要动手、怎么动手。
严格过滤噪音，宁缺毋滥。

## 1. 信源分级（决定是否纳入）

优先级：**T2 > T1 > T3**（可实践落地优先于反应速度）

- **T2 可复现实践**：附完整代码/配置/操作步骤的帖子、技术博客、GitHub 仓库。可信度看"能否照着做出来"。
- **T1 官方一手**：官方 changelog、官方文档更新、GitHub release notes、官方工程博客。本身即事实源，无需交叉验证。
- **T3 观点与传闻**：KOL 推文、YouTube 解读、无实操细节的评价类内容。**默认过滤**。仅当多个独立 T3 信源指向同一事件、且能回溯至 T1/T2 时才纳入。

**热度只用于同级内容排序，不用于决定纳入。**

## 2. 纳入白名单（命中其一才进报告）

1. **能力变化**：新模型、新功能、参数与行为变更（影响现有工作流的）
2. **一人团队可部署实操**：营销自动化、内容生产、招商线索处理、行研工作流、编程 agent 协作模式
3. **可靠性工程**：降低幻觉、长程任务稳定性、交付物验证方法、评估手段
4. **踩坑警报**：社区集中报告的 bug、性能退化、误导性宣传的辟谣

**排除项**（即使高热度也不进）：AI 行业投资/融资、股价变动、AI 机器人硬件、
与一人团队生产力无关的纯学术进展。

**加分项（不占精选名额，放扩展区）**：与充电场站/东南亚市场直接相关的 AI 营销、招商应用案例。

## 3. 置信标签

- ✅ **已证实**：官方来源直接确认
- 🔶 **可复现**：有完整步骤/代码，未经官方确认
- ⚠️ **待观察**：多信源提及但缺一手证据 → 明确标注"先别投时间"

## 4. 卡片结构（每条精选信息）

> 是什么 → 对我的场景值多少（明确到招商/Marketing 一人团队）→ 怎么部署（步骤或指向 T2 源）→ 坑与验证方法 → 行动建议（今天就试 / 观望一周 / 忽略）

## 5. 数量与版面

- 精选卡片：**单次运行 3-5 条**，宁缺毋滥。无达标信息时明确写"今日无值得行动的更新"。
- 扩展区：其他有价值但未入精选的条目，简要罗列，与精选区明确区隔。
- 今日噪音：1-2 条被过滤的高热度夸张内容 + 过滤理由。

**同日多次运行（追加制，每天一个入口）：**
- 同一日期若执行多次，第 N 次运行**不覆盖、不另建文件**，将新内容**追加写入当日已有 HTML 文件**。
- 追加位置：新精选卡片追加至精选区末尾，新扩展条目追加至扩展区末尾，新噪音条目追加至噪音区末尾。每次追加前插入时间戳分隔线（如 `── 第 2 次采集 · HH:MM（UTC+8）──`），区分批次。
- 页面顶部统计行同步更新（如"精选 5 条 · 本日已采集 2 次"）。
- **数量规则按单次运行计**：3-5 条是单次运行的质量锚，不是全天上限。第 1 次 3 条 + 第 2 次 2 条 = 页面累计呈现 5 条。
- 每次追加运行独立执行完整的信源分级、白名单筛选与交叉验证，且不得与当日已推内容重复（同事件有实质新进展除外，标注"更新"）。

## 6. 链接保留标准（每条 3-5 个）

优先级：
1. 可复现步骤最完整的实操源
2. 官方一手源
3. 含质疑与验证过程的高质量讨论串

不保留：营销号转述、无增量的二手总结。

## 7. 推送

- 每日 UTC+8 早 8:00 前生成完毕（当日追加运行不受此时限）
- 中文撰写，保留原文链接与关键英文术语
- HTML 发布至 GitHub Pages，**每天仅一个文件**：`docs/reports/YYYY-MM-DD.html`（UTC+8 日期）。同日追加运行更新同一文件，**不产生 `-2`、`-3` 等后缀文件**。
- docs/index.html 索引**每天仅一个入口**，描述随当日追加内容更新（含累计条数与采集次数）。
- 在对话中发送当日 HTML 文件
- 页脚标注当期 ROUTINE 版本号

## 8. 打磨机制

1. **随手反馈**：对话中任何反馈即触发规则修订并升版本号
2. **漏报归因**：用户提供漏报案例 → 逆向分析（信源未覆盖 / 规则误杀）→ 修规则
3. **每周复盘**：每周日报告末尾附本周"行动建议"命中率回顾
4. **变更记录**：所有规则改动记录于下方 Changelog，可审计、可回滚

## 9. 已推内容时效规则（滚动窗口）

**目标**：避免同一条信息反复占用精选名额，同时不漏掉仍在行动窗口内的重要更新。

**默认窗口：7 天。** 一条信息进入精选后，7 个自然日内不再重复精选，除非触发下列任一**再入选条件**：

1. **反转或重大修正**：新的 T1/T2 源与原精选结论直接矛盾（如官方撤回、已修复的 bug、定价调整）
2. **Deadline 临近**：原精选中有明确截止日期，且距截止 ≤ 3 天时，可重新入选并标注剩余天数
3. **实质性新进展**：有显著新 T1/T2 证据补充原有信息（如"观望一周"后有了明确结论）

**扩展区不受此限制**：已推内容的后续小更新可在扩展区用一行注明"后续更新"，不占精选名额。

**执行方式**：每次运行前，检索近 7 天已推报告的精选标题列表，标注窗口内的条目，再对比本次候选内容。

## 10. 信息采集深度规范

**目标**：消除不同模型/不同运行之间搜索深度不一致导致的漏报。

**社区来源定义（含社交媒体）**：Reddit（r/ClaudeAI 等）、Hacker News、X/Twitter（官方账号 + 工程师）、LinkedIn 行业讨论、Discord 官方频道、Substack/技术博客。

**最低要求（每次运行，指发起搜索的次数，非结果条目数）**：

- 独立搜索请求（WebSearch + WebFetch 合计）**不少于 6 次**，覆盖以下全部类型：

| 类型 | 次数 |
|------|------|
| 官方 changelog / release notes（优先 WebFetch 官方源，勿只靠搜索摘要） | 1-2 |
| 能力变更 + 新模型动态 | 1-2 |
| 社区实践 + 踩坑（含社交媒体） | 1-2 |
| 弃用 / 截止日期通知 | 1 |
| 社交媒体热议交叉验证 | 1 |
| 场景加分项（充电场站 / 东南亚 / 招商营销） | 1（可选） |

**追挖规则**：若 T3 来源提及某事件且命中纳入白名单，必须追加至少 1 次搜索寻找 T1/T2 确认；找不到则降级至扩展区并标 ⚠️ 待观察。

**不追挖的情形**：官方 T1 直接证实的条目、纯噪音条目（直接过滤）。
