1. 降本利器：Prompt Caching（读取 0.1×）+ Batch API（5 折）叠加，输入成本最高省 ~95%✅ 已证实

是什么：两个官方降本杠杆可叠加：① Prompt Caching——缓存读取仅 0.1× 基础输入价，5 分钟写入 1.25×、1 小时写入 2×；稳定内容（system prompt、工具定义、知识包）首次之后省 70-90%。② Batch API——异步批处理，输入输出各打 5 折。两者叠加时缓存读取在 batch 下有效低至 0.05×。以 Opus 4.8 为例：基础输入 $5/MTok，缓存读取降至 $0.50，叠加 batch 后约 $0.25。
对你的价值：直接压低你两类长期成本。① 本雷达系统 + 行研工作流有大量固定不变的规则文档、知识包、system prompt，正是缓存的理想对象——这些内容每次运行原样重发，加缓存后从第二次起省 90%。② 招商话术批量生成、竞对批量摘要这类不要求实时的任务走 Batch API，直接砍半。两者对一人团队的月度 API 账单影响显著。
怎么部署：① 缓存：在稳定前缀（system prompt / 规则文档 / 知识包）末尾加 "cache_control": {"type": "ephemeral"}；间隔较长的跟进用 "ttl": "1h"。② 批处理：非实时任务改走 /v1/messages/batches。③ 组合：批处理请求里同样可带 cache_control。
坑与验证：① 最小可缓存长度 512-4096 token（因模型而异），短 prompt 不会缓存；② 缓存写入有成本（1.25×/2×），只对高频复用内容划算，一次性内容别加；③ 2026-02 起缓存隔离改为 workspace 级，多 workspace 不共享缓存前缀。验证：对比加缓存前后账单的 cache_read_input_tokens 字段。
🎯 行动建议：今天就试——把你雷达/行研工作流里那份固定不变的规则文档或知识包加上 cache_control，跑两次看第二次的 cache_read_input_tokens，确认省下 90% 输入成本。
